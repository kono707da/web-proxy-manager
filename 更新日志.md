# 更新日志

## v1.0.8 - 2026-07-10

### 修复

- **passlib + bcrypt 5.x 不兼容导致创建默认管理员失败**：`passlib` 1.7.4 与 `bcrypt` 5.x 不兼容（`detect_wrap_bug` 阶段触发 `ValueError: password cannot be longer than 72 bytes`），移除 passlib 依赖，改用 bcrypt 原生 API（`bcrypt.hashpw` / `bcrypt.checkpw`）直接实现密码哈希与校验
- 从 requirements.txt 移除 `passlib[bcrypt]>=1.7`

## v1.0.7 - 2026-07-10

### 修复

- **容器启动失败 ModuleNotFoundError: No module named 'passlib'**：security.py 使用 `passlib.context.CryptContext` 做 bcrypt 密码哈希，但 requirements.txt 中缺少 `passlib[bcrypt]` 依赖，Docker 构建时未安装

## v1.0.6 - 2026-07-09

### 修复

- **Deploy 阶段端口冲突（port 8000 already allocated）**：hongkong 上 8000 端口被 etamusic 容器占用，proxy-manager 的 Web 界面宿主机端口改为 9000
  - Jenkinsfile：`docker run` 端口映射从 `-p 8000:8000` 改为 `-p 9000:8000`（容器内仍监听 8000，宿主机对外 9000）
  - docker-compose.yml：同步改为 `9000:8000`
  - README.md：端口说明、docker run 示例、访问地址全部更新为 9000
  - 成功提示改为 `访问 http://38.92.9.207:9000`
  - 代理端口 7890 不变

## v1.0.5 - 2026-07-09

### 修复

- **Deploy 阶段 SSH 执行返回 125 但无错误详情**：
  - 开启 `sshPublisher` 的 `verbose: true`，让远程命令 stdout/stderr 输出到 Jenkins 日志
  - 修正 `execCommand` 逻辑 bug：原 `A && B || true && C` 因运算符优先级会吞掉 `docker pull` 失败，改为括号明确分组 `(B || true)`，让 pull 失败时正确短路报错
  - 末尾追加 `|| (echo '=== DEPLOY FAILED, DIAGNOSTICS ===' && docker version && docker info && docker ps -a && exit 1)`，失败时打印 docker 版本/信息/容器列表，便于定位 125 根因（可能是 hongkong 无法访问内网 registry 或未配置 insecure-registries）

## v1.0.4 - 2026-07-09

### 修复

- **Docker 构建 apt purge 失败（exit code 100）**：`gzip` 是 Debian essential 包，`apt-get purge -y wget gzip` 不允许移除。改为 `apt-get purge -y --auto-remove wget`，只移除 wget 及其自动依赖，保留 gzip

## v1.0.0 - 2026-07-09

### 首次发布

基于 [mihomo (Clash.Meta)](https://github.com/MetaCubeX/mihomo) 内核的 Linux Docker 代理管理器，参考 [clash-verge-rev](https://github.com/clash-verge-rev/clash-verge-rev) 设计，通过 Web UI 进行流量控制。

### 新增功能

#### 后端（Python + FastAPI）

- **配置加载**：三层配置（config.yaml → 环境变量 `PROXY_MANAGER_` 前缀 → settings.json），敏感 secret 持久化到 data 目录
- **数据库**：SQLAlchemy 2.0，6 张表（User / Subscription / Rule / SpeedLimit / TrafficQuota / TrafficStat），首次启动自动创建默认管理员 admin/admin123
- **认证**：bcrypt 密码哈希 + JWT (python-jose)，手写 Header Bearer 依赖，支持 token 刷新
- **mihomo 内核管理**：
  - `MihomoClient`：封装 version / configs / connections / proxies / rules RESTful API
  - `config_builder`：订阅拉取与缓存、proxy-groups（select + url-test 自动测速）、限速写入（download-speed-limit / upload-speed-limit）、配额阻断规则（IP-CIDR,x.x.x.x/32,REJECT）自动注入、默认规则（GEOIP CN DIRECT + MATCH PROXY）
  - `MihomoManager`：子进程管理（start / stop / restart / reload）、流量轮询循环、配额检查循环（按客户端 IP 累计流量，超限触发配置热重载）
- **API 路由（9 个模块）**：
  - `/api/auth` —— 登录 / 刷新 / 当前用户
  - `/api/system` —— 内核启停 / 模式切换 / 日志级别 / 状态查询
  - `/api/subscriptions` —— CRUD + 立即更新 + 批量更新
  - `/api/proxies` —— 节点列表 / 分组 / 选择节点 / 测速
  - `/api/rules` —— CRUD（写操作触发配置热重载）
  - `/api/connections` —— 列表 / 关闭单个 / 关闭全部
  - `/api/traffic` —— 实时流量 / 内存 / 历史统计（可重置）
  - `/api/limits` —— CRUD（global / group / client 三种粒度）
  - `/api/quotas` —— CRUD + 重置使用量

#### 前端（Vue 3 + Pinia + reka-ui）

- **黑金主题**：Tailwind CSS + CSS 变量（gold #d4af37 + ink 深黑）
- **9 个页面**：
  - LoginView —— 全屏登录卡片
  - DashboardView —— 4 卡片概览 + 流量统计表 + 2 秒轮询
  - ProxiesView —— 分组卡片 + 节点表格 + 测速
  - SubscriptionsView —— 订阅 CRUD + 立即更新 / 批量更新
  - RulesView —— 规则 CRUD + 类型 / 目标选择器
  - LimitsView —— 限速 CRUD + scope 选择 + formatSpeed 预览
  - QuotasView —— 配额 CRUD + 使用率进度条 + 重置
  - ConnectionsView —— 连接列表 + 3 秒轮询 + 自动刷新开关 + 关闭连接
  - SettingsView —— 内核启停 + 模式切换 + 日志级别
- **认证**：Pinia Setup Store + localStorage 持久化，axios 请求拦截器注入 Bearer token，401 自动跳转登录
- **组件库**：完整 shadcn-vue（reka-ui）组件库 + ConfirmDialog + Toaster
- **路由守卫**：未登录自动跳转登录页，登录后回跳原路径

#### 部署与运维

- **Docker 多阶段构建**：Node 20 构建前端 → Python 3.12-slim 运行时 + 内置 mihomo 二进制（支持 build-arg 指定版本 / 架构 / 下载代理）
- **docker-compose.yml**：暴露 8000 (Web) + 7890 (代理混合端口)，数据卷持久化
- **Jenkinsfile**：符合 project-release.md 规范，sshPublisher + docker run 方式部署到 hongkong (38.92.9.207)
- **启动脚本**：start.bat（端口检查 + 前端构建检查 + mihomo 二进制检查 + 启动后端）+ kill_stale_port.ps1（本应用占用则杀掉重启，他人占用则提示）

### 端口说明

|---|---|
| 8000 | Web 管理界面（前后端合并） |

- 前端：Vue 3.4 + Pinia 2.1 + reka-ui 2.10 + Tailwind CSS 3.4 + axios
- 部署：Docker 多阶段构建 + docker-compose + Jenkins CI/CD
